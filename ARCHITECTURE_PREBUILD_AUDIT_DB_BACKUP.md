# Pre-Build Audit — `audit.db` Backup & Restore Strategy (narrow scope)

> Status: **AUDIT ONLY.** No code, no implementation prompts. Scope is deliberately
> narrow, per request: backup and restore for `audit.db` only — not a general backup
> strategy for the library images, not a Shopify sync-state store, not anything else
> named in the Phase 3 closure ADR's prerequisite list.

---

## 1. Current State

`audit.db` is a single SQLite file, preserved by default since the database-preservation
fix, living on one external drive with no redundancy of any kind. It now holds data in
three distinct categories with three distinct recovery profiles:

- **Fully re-derivable** — identity (families, designs, assets, products, SKUs). Lost, it
  can be rebuilt exactly by re-running the importer against the image library. Zero real
  risk.
- **Re-derivable at a cost** — local colour extraction. Free to regenerate, but represents
  real time already spent.
- **Non-derivable** — `vision_results` (real, paid AI analysis) and `display_title`
  manual overrides (human review decisions). This is the category that matters. It cannot
  be regenerated from the image library; it can only be regenerated by spending money
  again (AI) or redoing human judgment (title acceptance) — and in the conflict cases
  reviewed this session, that judgment involved actually opening and comparing source
  images, not a trivial re-decision.

**One relevant fact from this session specifically:** the recent `git` commit and push do
**not** protect this data. The `.gitignore` set up alongside that commit deliberately
excludes `reports/`, and therefore `audit.db`, from version control — correctly, since a
growing SQLite binary doesn't belong in git history, but this means the commit that just
happened backs up the *code*, not the *data*. If asked "didn't we just back this up by
pushing to GitHub," the honest answer is no.

**Technical detail that shapes the whole design:** the database runs in **WAL
(write-ahead log) mode** (confirmed directly in `model.py`). Under WAL, recently committed
data can live in a separate `audit.db-wal` file alongside `audit.db` itself, not yet
folded into the main file. A naive "copy the `.db` file" backup — a Finder drag-copy, a
plain `cp` — can produce a backup that is silently missing recent writes, or in worse
cases is internally inconsistent. This is not a hypothetical: it is the specific,
well-documented failure mode of backing up a WAL-mode SQLite database by file-copy alone,
and it is exactly the kind of thing that looks fine until the day someone actually needs
to restore it.

## 2. Risks

**Critical**
- **No backup exists at all**, for data already proven non-derivable and already
  representing real spend and real human review time (ten accepted titles, real AI
  analysis, as of this session). Single point of failure: one external drive.
- **A naive file-copy backup approach would be actively unsafe, not just absent.** Given
  WAL mode, the "obvious" fix — periodically copying `audit.db` — is itself a risk if not
  done correctly. Whatever is built must account for this by construction, not as an
  afterthought.

**Major**
- **`--fresh` is the single most dangerous operation already in the tool**, and it has
  already caused real data loss once (via the prior default-delete behaviour, now fixed).
  It still exists, correctly, as an explicit destructive option — but nothing today
  protects against a moment of carelessness before running it. This is the one place a
  backup step would pay for itself immediately, almost regardless of what else is or
  isn't built.
- **No restore procedure has ever been exercised.** A backup that has never been proven
  to restore correctly is not a trustworthy backup — this project's own discipline (prove
  the schema migration against a real reproduction, not just design it) should apply here
  with at least equal force, since data loss is a worse failure mode than a crash.

**Minor**
- No retention/rotation policy exists or has been decided (how many backups, for how
  long, on what schedule).
- No decision yet on *where* backups should live relative to the source drive — same
  drive (protects against accidental deletion or corruption of the live file, not against
  drive failure), a second physical drive, or cloud storage (protects against total local
  loss) each cover different failure modes and are not interchangeable.
- No automation exists (manual on-demand vs. scheduled) — not urgent to decide immediately,
  but worth deciding deliberately rather than defaulting silently.

**Observation**
- The existing schema-migration system is a genuine asset here: a backup taken under an
  older schema version will still open correctly under newer code (that's exactly what the
  migration system proved), so backups don't need to be schema-version-pinned or
  regenerated whenever the code changes.
- Python's standard library `sqlite3` module provides a built-in, WAL-safe backup API
  (a hot-backup mechanism designed for exactly this situation) rather than requiring a
  shell-level file copy — the natural, already-available building block for whatever gets
  implemented, not a new dependency.

---

## 3. Findings

**What already exists:** nothing specific to backup or restore. What *does* already exist
and is directly relevant: the preserve-by-default database lifecycle, the schema migration
system (which makes old backups forward-compatible with newer code), and the precedent of
treating `--fresh` as the one operation requiring extra care (it already prints a
destructive-action warning).

**What's missing:** any backup mechanism at all; any restore procedure; any point where a
backup happens automatically around the one operation known to be destructive; any
retention policy; any decision about backup location relative to the source drive.

**Hidden coupling:** none of consequence. A backup mechanism is naturally isolated — it
reads the database file (or calls a proper backup API), it doesn't need to understand
identity, vision, or SKU logic at all. This mirrors how cleanly the schema migration
system was added without touching unrelated code.

**Reuse potential:** high. The most natural, lowest-risk integration point is inside
`build_graph.py`'s existing `--fresh` handling — since that code path already exists,
already knows it's about to be destructive, and already prints a warning before acting.
An automatic safety snapshot immediately before a `--fresh` wipe would directly close the
single most dangerous gap named above, using infrastructure that's already in place.

---

## 4. Recommendation

**Approved — scoped narrowly, mirroring how 3-d.2 was right-sized against real need
rather than hypothetical scale.**

The audit does not recommend building a general-purpose, scheduled, multi-location,
cloud-synced backup system. That would be scope ahead of evidence, the same mistake the
3-d.2 audit corrected for the acceptance workflow. The evidence here supports something
much smaller:

1. A **WAL-safe backup mechanism** (using the standard library's proper backup API, not a
   raw file copy) that produces a timestamped, complete snapshot on demand.
2. An **automatic safety snapshot immediately before `--fresh` executes** — the one
   integration point that directly addresses the Major risk above, at essentially zero
   added complexity since the destructive code path already exists and already gates on
   a warning.
3. A **restore procedure that is actually exercised and proven** before this is considered
   done — not just documented, in keeping with this project's standing discipline of
   proving fixes against real reproductions rather than trusting the design.

Retention/rotation policy and off-drive/cloud storage are explicitly **not** decided by
this audit — they depend on facts about your actual environment (is Time Machine already
running on this Mac? is there a second drive or a cloud-synced folder already in use?)
that should be answered before, not during, implementation.

---

## 5. Acceptance Criteria (for whichever implementation is approved next)

- Backups are produced via a WAL-safe mechanism — verified correct even immediately after
  a write, with no dependence on the database being idle or checkpointed first by hand.
- A backup taken, then restored, produces a database that is byte-for-byte equivalent in
  content to the source at the moment of backup — proven with a real test that writes
  data, backs up, corrupts or discards the original, restores, and confirms nothing was
  lost. This mirrors exactly how the schema migration fix was proven against a real
  old-schema database, not just asserted to work.
- `--fresh` automatically produces a safety snapshot before wiping, without requiring the
  operator to remember a separate step.
- The backup mechanism does not require the database to be closed, idle, or otherwise
  taken offline — it must be safe to run against a live, in-use database.
- No change to identity, SKU, vision, manifest, or Shopify code — this is purely additive
  infrastructure around the database file's lifecycle.
- Existing regression remains green; new tests prove the backup/restore round-trip
  specifically, including a case with uncommitted WAL data present at backup time.

## Open Questions to Resolve Before Implementation

Unlike most of this project's decisions, these depend on facts about your actual
environment, not architectural judgment — worth answering before scoping the build:

1. **Does this Mac already have Time Machine (or another whole-disk backup) running**,
   and does it include the `Work_4TB` external drive? If yes, that may already provide a
   second copy for free, and the new mechanism mainly needs to produce a *restorable*
   snapshot Time Machine can pick up, rather than solving off-drive storage itself.
2. **Is there a second physical drive, or a cloud-synced folder (Google Drive, Dropbox,
   iCloud) already in regular use** that a backup could be pointed at, or does one need to
   be set up specifically for this?
3. **How often is `audit.db` likely to change** going forward (occasional AI batches and
   accepts, vs. frequent daily use)? This determines whether on-demand backup is
   sufficient or a schedule is worth adding later.

---

## Deliverables Summary

**Current State:** no backup exists; the database now holds non-derivable, paid, and
human-reviewed data; WAL mode makes naive file-copy backups actively unsafe rather than
merely insufficient; the recent git push does not cover this data.

**Risks:** Critical — no backup, and naive backup would itself be unsafe under WAL.
Major — `--fresh` is unprotected despite being the tool's one proven-dangerous operation;
no restore has ever been tested. Minor — no retention policy, no location decision, no
schedule. Observation — the schema migration system and Python's built-in WAL-safe backup
API both make this cheaper to build correctly than it might first appear.

**Recommendation: Approved, narrowly scoped** — a WAL-safe on-demand backup mechanism, an
automatic safety snapshot before `--fresh`, and a proven (not just designed) restore path.
Retention policy and off-drive storage deferred pending the environmental questions above.

**Next step:** answer the three open questions, then this becomes a small, well-bounded
implementation — comparable in size and risk to the schema migration fix, not a new
subsystem.
